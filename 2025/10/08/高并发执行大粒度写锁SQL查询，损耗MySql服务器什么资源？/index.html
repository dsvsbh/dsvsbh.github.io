高并发执行大粒度写锁SQL查询，损耗MySql服务器什么资源？

发表于2025-10-08|更新于2025-10-08
|浏览量:

问题

在面试过程中被问到一个问题：

高并发下执行SQL语句：select * from t for update ，会损耗MySql服务器的什么资源？

思考

  1. for update 语句是对查到的记录进行加写锁（行级锁），而这个查询语句并没有where查询条件，说明这个语句对整个表记录上了写锁。
  2. 在高并发执行这个语句的时候会导致大量事务因为没抢到这个锁而阻塞，问题转化为了：服务器收到高并发请求，并且大量请求处理出现阻塞，服务器什么资源会出现瓶颈？

解答

这是一个非常好的问题，它触及了系统性能和稳定性的核心。当一个节点的QPS很高，并且这些请求发生了阻塞，会从多个层面快速损耗节点的关键资源，最终导致服务不可用。

简单来说，最直接损耗的资源是线程，但会像多米诺骨牌一样，连锁消耗以下四大类资源：

  • ​线程资源​
  • 内存资源​
  • ​CPU 资源​
  • ​网络/文件描述符资源​

下面我们详细拆解这个过程。

1. 线程资源 - 最直接的损耗

​机制​：在典型的Web服务器（如Tomcat、Netty）或应用框架（如Spring MVC）中，通常使用线程池来处理并发请求。每个进来的请求都会由一个工作线程来处理。

​损耗过程​：
当请求处理被阻塞时（例如，等待数据库响应、调用缓慢的下游服务、获取锁等），这个工作线程无法被释放。它虽然不占用CPU，但它会一直被这个请求占用，处于等待状态。

如果阻塞的请求持续涌入，线程池中的线程会被一个个地占用。当所有线程都被阻塞占满时，线程池就耗尽了。

​后果​：​线程池耗尽。此时，新的请求无法得到线程来处理，服务器会开始拒绝请求（返回5xx错误，如503 Service Unavailable）。这是高QPS伴随阻塞时最先出现、也最典型的故障现象。

2. 内存资源 - 紧随其后的损耗

​机制​：每个活跃的线程都有其独立的线程栈，需要占用一定的内存（例如，在JVM中，可通过 -Xss参数设置栈大小，通常默认是1MB）。更重要的是，每个请求通常都会携带一些数据（如请求参数、Session信息、数据库连接对象等），这些数据会作为对象分配在堆内存中。

​损耗过程​：

​线程栈内存​：大量被阻塞的线程意味着有大量几乎不工作的线程存在，但它们每个都占用了不小的栈内存。1000个被阻塞的线程就可能占用近1GB的栈内存。

​堆内存​：每个被阻塞的请求，其关联的请求和响应对象会长时间无法被回收。随着阻塞请求的堆积，会有大量临时对象堆积在堆中，导致内存使用率飙升。

​后果​：​内存溢出。特别是JVM的堆内存被占满后，会频繁触发Full GC。如果GC无法回收这些因阻塞而无法完成处理的对象，最终会抛出 OutOfMemoryError，导致整个应用进程崩溃。

3. CPU 资源 - 间接但严重的损耗

​机制​：CPU本身并不直接消耗于“阻塞”，阻塞的线程会让出CPU。但高并发阻塞会通过以下方式间接消耗大量CPU：

​大量的上下文切换​：当有成千上万个线程（大部分处于阻塞状态）时，操作系统的线程调度器需要不断地在可运行的线程之间进行切换，这个过程（上下文切换）本身是非常消耗CPU的。

​频繁的垃圾回收​：如上所述，内存压力会引发频繁的GC，而GC（尤其是Full GC）是一个CPU密集型操作。系统本应用来处理业务的CPU周期，被大量浪费在GC和上下文切换上。

​后果​：​CPU利用率飙升，但系统吞吐量极低。你会发现CPU使用率很高，但实际处理的请求数（TPS）却很低。系统响应变得极其缓慢。

4. 网络连接与文件描述符 - 底层资源的损耗

​机制​：每个网络连接、每个打开的文件都会占用一个文件描述符。操作系统对单个进程能打开的文件描述符数量是有限制的。

​损耗过程​：由于请求处理被阻塞，TCP连接无法被及时关闭。新的请求又在不断建立连接，导致连接数和打开的文件描述符持续增长。

​后果​：​文件描述符耗尽。当达到上限时，操作系统会拒绝进程建立新的连接或打开文件，表现为无法接受新请求、日志打不开等，服务彻底僵死。

如何应对与缓解？

  • ​优化阻塞点​：从根本上解决问题，优化慢SQL、缓存结果、使用更快的下游服务。
  • ​设置超时与重试​：为所有外部调用设置合理的超时时间，避免无限期等待。并配合合理的重试策略（如退避重试）。
  • ​限流​：在入口层面进行限流，拒绝超出系统处理能力的请求，保护节点不被打垮。常用算法有令牌桶、漏桶。
  • ​熔断​：当下游服务不可用或响应过慢时，主动熔断对其的调用，直接快速失败，避免线程资源被耗尽。
  • ​异步化​：使用异步非阻塞的编程模型（如Reactor、WebFlux），用少量线程处理大量并发连接，从根本上避免“一个线程一个请求”的阻塞模型。
  • ​扩容与降级​：水平扩展节点数量，或在系统压力大时，关闭非核心功能，保障核心链路畅通。

问题解决

针对原问题解决方案：

  1. 降低锁粒度：如果业务并不需要全表数据，可以加上where条件筛选，然后对查到的记录上锁，防止锁住全表导致其他所有事务阻塞。
  2. 降低锁级别：如果查询只是想要防止在读的过程中数据被篡改，可以将写锁降级为读锁，这样其他事务也可以拿到读锁进来读数据，只会阻塞写操作。
  3. 分库分表：当数据量很大的时候可以考虑分库分表，这样多个库表之间的事务是可以并行的，可以降低对单个数据库或者表的压力。
